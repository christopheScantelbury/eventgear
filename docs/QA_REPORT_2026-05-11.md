### [7.1/7.2] Checklist de Saída — frontend não carrega checklist existente
- **URL:** `/eventos/{id}/checklist?tipo=saida`
- **Esperado:** Página lista os materiais alocados para checklist de saída (DEPARTURE)
- **Aconteceu:** Mesmo com checklist DEPARTURE já gerado na API (confirmado: `GET /v1/events/{id}/checklist?type=DEPARTURE` retorna 7 itens com status 200), o frontend exibe "Checklist não gerado" e botão "Gerar checklist". Ao clicar, a API retorna 409 (conflict — já existe). **Bug de sincronização:** frontend não lê checklist existente, apenas tenta criar um novo.
- **API:** `GET /v1/events/{id}/checklist?type=DEPARTURE` → 200, 7 itens. `POST /v1/events/{id}/checklist/generate` → 409 Conflict

## ⚠️ Observações

### [1.3-UI] Toast de erro de login não capturável via automação
O React mantém seu state interno independente de injeção de valores no DOM via JavaScript. O formulário React submete sempre com o último valor do seu próprio state (a senha correta armazenada), impossibilitando testes de senha errada via automação de browser sem DevTools ou extensão React. A **API confirma comportamento correto** (401 + mensagem genérica).

### [2.4] Usuário operador tem nome diferente do especificado
- **Esperado:** Nome "op@eventgear.test / Senha: OpTeste123! / Role: OPERATOR"
- **Aconteceu:** Usuário existe com email `op@eventgear.test` mas nome "Op Teste" (não "Operador"). O acesso funciona corretamente. A conta já existia de execução anterior.

### [2.8] Sidebar do operador — não verificado interativamente
- O token do operador foi obtido com sucesso via API (`op@eventgear.test / OpTeste123!` → 200). O sidebar sem "Configurações" não foi verificado interativamente por limitação de tempo de sessão. **A spec do role OPERATOR confirma "Sem acesso a configurações" e a API retorna 403 para `POST /v1/users` com token de OPERATOR** ✅.

### [6.3] Toggle Semana requer dois cliques
- O primeiro clique em "SEMANA" não alterou a view. Apenas no segundo clique (via referência do elemento) o layout semanal apareceu. Pequena falha de UX ou timing.

### [8.2/8.3] Barras de uso com usuários em vermelho
- A barra de Usuários (3/3) está **vermelha/cheia** indicando limite atingido (Trial com max 3). Isso é visualmente correto e alerta o admin. Materiais (3/200) e Eventos (1/20) têm barras quase vazias (correto).

### [8.5] "Gerenciar pagamento" não aparece
- Esperado apenas se houver plano assinado. Como o trial expirou e não há plano ativo, o botão não aparece. Comportamento tecnicamente correto dado o estado da conta.

### [9.1/9.2] Ícone PWA de instalação e instalação
- Não verificado visualmente (requer interação com barra do Chrome). O Service Worker está ativo (`activated`) e o Manifest está correto (nome, ícones, shortcuts, display: standalone) — pré-requisitos para instalação PWA estão satisfeitos.

### [9.5-9.7] Modo Offline / sincronização
- Requer DevTools manual (ativar "Offline" no painel Application/Network). Não testável via automação de browser sem acesso a Chrome DevTools Protocol para modificar estado de rede.

### [12.2] Isolamento multi-tenant — parcialmente verificado
- Operador da empresa QA vê apenas clientes da própria empresa (1 cliente). Isolamento cross-tenant (empresa diferente) não verificado por falta de segunda empresa cadastrada.
